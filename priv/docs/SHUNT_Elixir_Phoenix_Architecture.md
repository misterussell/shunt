# SHUNT — Elixir/Phoenix Architecture Specification
## Engineering Handoff Document

Version: 1.0

---

# Architectural Philosophy

SHUNT should be built as a content-driven simulation running on the BEAM.

Goals:

- Content-first development
- Minimal database reads during gameplay
- Clear separation between simulation and UI
- Event-driven progression
- Scalable to thousands of concurrent players
- Easy content authoring

The architecture should treat Phoenix LiveView as a presentation layer, not the game engine.

---

# High-Level Architecture

```text
Browser
    ↓
Phoenix LiveView
    ↓
Game API Layer
    ↓
Game Systems
    ↓
Persistence Layer

Content Registry
World State
Event Engine
Effect Engine
```

LiveView should never contain gameplay logic.

---

# Core Runtime Systems

## Content Registry

Purpose:

Load static game content into memory.

Examples:

- Locations
- Opportunities
- Events
- Recipes
- NPC definitions
- Faction definitions

Recommended storage:

```elixir
ETS
```

Alternative:

```elixir
:persistent_term
```

Use ETS if hot reloading content is desired.

Use persistent_term if content is truly static.

---

## World State Service

Purpose:

Track global simulation state.

Examples:

- Active market conditions
- Global faction conflicts
- Economic modifiers
- Scheduled events

Recommended:

```elixir
GenServer
```

Responsibilities:

- Maintain active modifiers
- Rotate market conditions
- Broadcast changes

---

## Event Engine

Purpose:

Execute gameplay content.

Flow:

```text
Opportunity
    ↓
Select Event
    ↓
Present Choices
    ↓
Resolve Choice
    ↓
Generate Effects
```

The Event Engine should not mutate state directly.

---

## Effect Engine

Highest-priority system.

Purpose:

Apply effects generated by gameplay.

Example:

```elixir
{:scrip, 150}
{:heat, 5}
{:skill_xp, :ghostwork, 50}
```

Responsibilities:

- Validate effects
- Apply effects
- Emit events
- Trigger unlock checks

Every state change should pass through the Effect Engine.

---

# Character Architecture

## Character Aggregate

Persistent player state.

Example:

```elixir
%Character{
  id: id,

  resources: %{},
  skills: %{},
  faction_rep: %{},
  npc_loyalty: %{},
  flags: MapSet.new(),
  unlocks: MapSet.new()
}
```

Store in PostgreSQL.

Load when player session starts.

---

## Session Process

Each connected player receives:

```elixir
CharacterServer
```

Recommended:

```elixir
DynamicSupervisor
```

Responsibilities:

- Hold active state
- Handle gameplay requests
- Cache character data
- Batch persistence

---

# Persistence Strategy

## Database Responsibilities

Persist:

- Character data
- Inventory
- Skills
- Reputation
- Flags
- Unlocks
- Crafting progression

Use PostgreSQL.

---

## ETS Responsibilities

Store:

- Static content
- Read-heavy lookup data
- Event definitions
- Recipe definitions

Never store authoritative player data in ETS.

# Content Pipeline

Recommended structure:

```text
priv/content/

locations/
events/
opportunities/
recipes/
npcs/
factions/
market_conditions/
```

Example:

```text
events/

underbelly/
    suspicious_cargo.exs
    ka_sting.exs

midgrid/
    corp_audit.exs
```

---

# Event Definition Pattern

Example:

```elixir
%Event{
  id: "suspicious_cargo",
  text: "...",
  choices: [...]
}
```

Events should remain data.

No embedded logic.

---

# Opportunity Definition Pattern

Example:

```elixir
%Opportunity{
  id: "inspect_shipment",
  requirements: [...],
  event_pool: [...]
}
```

The Event Engine chooses from the pool.

---

# Requirements Engine

Dedicated system.

Purpose:

Determine whether content is available.

Examples:

```elixir
{:skill, :ghostwork, 2}
{:flag, :met_vex}
{:rep, :syndicate, :trusted}
```

All content visibility should use requirement checks.

---

# Unlock System

Purpose:

Reveal new content.

Examples:

```elixir
{:unlock_location, "midgrid_exchange"}
{:unlock_opportunity, "corp_data_heist"}
```

Unlocks should be persistent.

---

# Market Architecture

## Market Service

Recommended:

```elixir
GenServer
```

Responsibilities:

- Rotate conditions
- Generate modifiers
- Broadcast changes

Example:

```elixir
Electronics Boom
Augmentation Shortage
Data Leak
```

---

# Heat Architecture

Heat should remain character-specific.

Do not create separate processes.

Store on character.

Heat should influence:

- Event weighting
- Risk calculations
- Opportunity availability

---

# LiveView Architecture

LiveView responsibilities:

- Render UI
- Receive player actions
- Subscribe to updates
- Display results

LiveView should not:

- Resolve events
- Execute effects
- Perform simulation

Flow:

```text
LiveView
    ↓
CharacterServer
    ↓
Event Engine
    ↓
Effect Engine
    ↓
CharacterServer
    ↓
LiveView Update
```

---

# PubSub Usage

Use Phoenix PubSub for:

- Market updates
- Character updates
- World events

Example:

```elixir
Phoenix.PubSub.broadcast(
  PubSub,
  "character:#{id}",
  {:character_updated}
)
```

---

# Scaling Strategy

## Early Stage

Single node.

Use:

- PostgreSQL
- ETS
- DynamicSupervisor

This is sufficient for launch.

---

## Later Stage

Introduce:

- Distributed PubSub
- Multi-node clustering
- Presence
- Regional simulation processes

Do not optimize for this initially.

---

# Critical Decisions

1. LiveView is presentation only.
2. CharacterServer owns active character state.
3. Event Engine generates outcomes.
4. Effect Engine mutates state.
5. Content lives outside engine code.
6. ETS stores content, not player data.
7. PostgreSQL remains authoritative.
8. Requirements and unlocks are generic systems.
9. Market state modifies probabilities.
10. Every new feature should integrate through Effects.
